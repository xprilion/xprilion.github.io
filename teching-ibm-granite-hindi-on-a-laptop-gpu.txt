16:T257a,<p>I've been on a bit of a small-model kick lately. Between the OpenAI Parameter Golf challenge and building browser agents with sub-1B models, I've developed a mild obsession with squeezing everything possible out of tiny GPUs. So when IBM dropped <a href="https://huggingface.co/ibm-granite/granite-4.1-3b" rel="noopener noreferrer">Granite 4.1 3B model</a> with genuinely good multilingual pretraining - I had one question: can I make it speak Hindi on my RTX 3070 Laptop?</p>
<blockquote>
<p>Spoiler: yes. But the road there involved two CUDA OOMs, one broken checkpoint resume, a GGUF export that refused to build, and a lot of <code>nvidia-smi</code> staring.</p>
</blockquote>
<h2>The Setup</h2>
<p>Here's what I was working with:</p>
<ul>
<li><strong>GPU</strong>: NVIDIA RTX 3070 Laptop (8 GB VRAM)</li>
<li><strong>Base model</strong>: <code>ibm-granite/granite-4.1-3b</code> (3.5B params, Apache 2.0). <a href="https://huggingface.co/ibm-granite/granite-4.1-3b" rel="noopener noreferrer">Link</a></li>
<li><strong>Dataset</strong>: <code>FreedomIntelligence/evol-instruct-hindi</code> — ~59K Hindi instruction samples in ShareGPT format. <a href="https://huggingface.co/datasets/FreedomIntelligence/evol-instruct-hindi" rel="noopener noreferrer">Link</a></li>
<li><strong>Method</strong>: QLoRA (4-bit NF4 quantization + LoRA adapters)</li>
<li><strong>Framework</strong>: Unsloth + SFTTrainer (TRL)</li>
</ul>
<p>The constraint was hard: 8 GB VRAM. </p>
<h2>Attempt #1: CUDA Go Brrr</h2>
<p>I started with what felt like conservative hyperparams:</p>
<pre><code class="language-python">max_seq_length = 1024
per_device_train_batch_size = 2
lora_r = 16, lora_alpha = 32
</code></pre>
<p>CUDA laughed at me. OOM within seconds.</p>
<p>Full fine-tuning a 3B model needs ~25 GB VRAM in fp16. Even with 4-bit QLoRA, the math isn't free - activations, optimizer states, and gradients all want their share. The 1024-token context length was the biggest memory hog; attention scales quadratically.</p>

<p>Effective batch size stayed at 8. The other tricks that helped:</p>
<ul>
<li><code>PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True</code> - prevented fragmentation OOMs</li>
<li><code>gradient_checkpointing=True</code> - traded compute for memory</li>
<li><code>optim="adamw_8bit"</code> — 8-bit optimizer states</li>
<li>Disabled <code>torch.compile</code> — saved 1-2 GB and avoided some Unsloth compatibility issues</li>
</ul>
<p>Here's the final training config:</p>
<pre><code class="language-python">from unsloth import FastLanguageModel

model, tokenizer = FastLanguageModel.from_pretrained(
    model_name="ibm-granite/granite-4.1-3b",
    max_seq_length=512
)

model = FastLanguageModel.get_peft_model(
    model,
    r=8, lora_alpha=16,
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj",
                    "gate_proj", "up_proj", "down_proj"],
    lora_dropout=0.05,
    use_gradient_checkpointing="unsloth",
)
</code></pre>
<h2>Why 400 steps? Why not 500?</h2>
<p>First real training run: 400 steps, loss dropped from <strong>1.28 → 0.53</strong>. Nice curve, no divergence. I ran perplexity eval on a held-out Hindi set:</p>

<tr>

<td><strong>0.53</strong></td>
</tr>

<p>74.7% reduction in perplexity. The model was actually generating coherent Hindi and answering factual questions, writing poetry, explaining Python loops. It retained its English and code abilities too, which is the whole point of LoRA.</p>
<p>I figured: <em>let's push it to 500 steps, see if we can squeeze out a bit more.</em></p>
<p>So I tried to resume from the checkpoint. And hit this gem:</p>
<blockquote>
<p>RuntimeError: torch.load with weights_only=True requires torch 2.6+</p>
</blockquote>
<p>Transformers 5.5 introduced a security fix that broke checkpoint loading on my PyTorch 2.5 install. The checkpoint was perfectly valid -  I just couldn't load it without upgrading half my stack, which would have likely broken Unsloth's compiled kernels.</p>
<p>Fine. Restart from scratch with <code>--max_steps 500</code>.</p>
<p>The second run followed almost the exact same loss curve. By step 400, loss was at ~0.57. By step 500, it plateaued at <strong>0.56</strong>. The model had saturated - pushing further wouldn't have helped.</p>
<h2>What Failed (And Why It's Fine)</h2>
<h3>GGUF Export</h3>
<p>Unsloth's <code>model.save_pretrained_gguf()</code> needs to compile llama.cpp from source. That needs <code>libssl-dev</code>, <code>libcurl4-openssl-dev</code>, and a bunch of other <code>-dev</code> packages. My environment didn't have sudo. I could've set up a Docker container or used a cloud VM, but honestly, the merged 16-bit safetensors model works perfectly with Transformers and loads in 4-bit with bitsandbytes. The GGUF would've been nice for Ollama/llama.cpp, but it wasn't blocking anything.</p>
<h3>8B Model: Not Happening</h3>
<p>I briefly considered trying the same recipe on Granite 8B. The math: even with 4-bit QLoRA at <code>seq_len=512</code>, it'd push ~10.5 GB VRAM. My 8 GB card said no. That's a cloud GPU job for another day.</p>
<h3>torch.compile</h3>
<p>Unsloth + torch.compile had some weird interactions on my CUDA 12.1 setup. Disabling it was simpler and only cost a few percentage points of throughput. On an 8 GB card doing QLoRA, you're bottlenecked by memory bandwidth anyway, not compilation overhead.</p>
<h2>What Actually Worked</h2>
<h3>Unsloth ✨</h3>
<p><strong>Unsloth is genuinely magic.</strong> It handled all the 4-bit loading, LoRA patching, and gradient checkpointing with a couple lines of code. The <code>FastLanguageModel.get_peft_model()</code> call auto-detects the architecture and sets up efficient kernels. Training throughput was ~0.6 samples/second, which got me through 500 steps in about 1.8 hours.</p>
<h3>Dataset choice</h3>
<p><strong>The dataset matters more than hyperparams.</strong> <code>FreedomIntelligence/evol-instruct-hindi</code> has 59K diverse instructions - factual QA, creative writing, code explanation, translation. The diversity meant the model learned Hindi <em>generation</em> rather than just Hindi <em>memorization</em>. I tried a smaller dataset initially and got word-salad output.</p>
<h2>Wandb 💛</h2>
<p><strong>W&amp;B tracking saved my sanity.</strong> I logged every 5 steps with <code>report_to="wandb"</code>. Being able to watch the loss curve in real-time (and notice it plateauing at step 400) meant I didn't waste hours training a saturated model.</p>
<h2>The Model in Practice</h2>
<p>After merging the LoRA adapter into the base model, I uploaded it to HuggingFace at <a href="https://huggingface.co/xprilion/granite-4.1-3b-hindi-lora" rel="noopener noreferrer">https://huggingface.co/xprilion/granite-4.1-3b-hindi-lora</a>. Loading it is straightforward:</p>
<pre><code class="language-python">from transformers import AutoModelForCausalLM, AutoTokenizer

model = AutoModelForCausalLM.from_pretrained(
    "xprilion/granite-4.1-3b-hindi-lora",
    device_map="auto"
)
tokenizer = AutoTokenizer.from_pretrained("xprilion/granite-4.1-3b-hindi-lora")

prompt = "मुझे पायथन में फॉर लूप समझाओ"
inputs = tokenizer(prompt, return_tensors="pt").to("cuda")
outputs = model.generate(**inputs, max_new_tokens=1000, temperature=0.7)
print(tokenizer.decode(outputs[0], skip_special_tokens=True))
</code></pre>
<p>Qualitatively, it handles:</p>
<ul>
<li>Factual QA: "भारत की राजधानी क्या है?" → correct, natural Hindi</li>
<li>Code explanation: Explains Python concepts in Hindi with code examples</li>
<li>Creative writing: Generates poetry, stories in Hindi</li>
<li>English retention: Still answers English prompts correctly</li>
</ul>
<p>It's not perfect. Complex multi-turn reasoning can drift, and very domain-specific Hindi (legal, medical) sometimes produces mixed-language output. But for a 3B model trained in under 2 hours on a laptop GPU? I'll take it.</p>
<h2>Conclusion</h2>
<p>Turns out your humble 8GB laptop GPU isn’t useless - it just has a very strong personality and demands compromises. With the right tricks, you can absolutely teach an old monkey a few new tricks 3B model new languages… just don’t expect it to handle too many tokens.. In the end, it’s less “brute force AI” and more “carefully negotiated truce with VRAM.”</p>
