18:T2849,<p>If you like shopping research but hate <em>tab debt</em>, this project is for you: <strong>search YouTube, pick a handful of videos, then ask questions and generate a course-like buying guide</strong>.</p><p>You can try the deployed version here:<a href="https://yt-shopping-ai.xprilion.com" rel="noreferrer"> yt-shopping-ai.xprilion.com.</a></p><p>This post is intentionally <em>not</em> a step-by-step beginner tutorial. It’s a build note about the decisions: <strong>why these choices</strong>, what they buy you, what they cost you, and when you’d pick the alternatives in real projects.</p><h2 id="the-problem">The Problem</h2><p>Shopping content on YouTube is great… and also chaotic:</p><ul><li>The best advice is often buried at minute 14:37 after three sponsor segues.</li><li>Different creators disagree (sometimes loudly) and the truth is in the overlap.</li><li>You don’t want “a summary”. You want <strong>a decision</strong>: what to buy, why, and the tradeoffs.<br></li></ul><p>So the product goal becomes:</p><blockquote><em>Turn a messy pile of video opinions into a compact, source-backed guide that helps a human choose.</em></blockquote><h2 id="what-you%E2%80%99re-building"><strong>What you’re building</strong></h2><p>A Next.js App that:</p><ul><li>searches for YouTube videos via Google’s search stack</li><li>lets you select videos</li><li>generates a structured “course outline”</li><li>extracts transcript-like summaries with timestamps</li><li>answers questions with sources back to timestamps</li></ul><p>And yes, it’s “bring your own keys” (BYOK), but that’s more of an operational choice than the main character.</p><h2 id="run-it-locally-minimal">Run it locally (minimal)</h2><p>If you already live in the Next.js ecosystem, you know the drill. Clone the repo, install deps, start dev.</p><p>If you <em>don’t</em> (or you’re rusty), here are the canonical references:</p><ul><li>Node install: <a href="https://nodejs.org" rel="noreferrer">https://nodejs.org</a></li><li>pnpm: <a href="https://pnpm.io/installation" rel="noreferrer">https://pnpm.io/installation</a></li></ul><p>Now, let's explore some of the decision making that went into building this project.</p><h2 id="decision-1-search-is-a-product-feature-not-a-backend-detail"><strong>Decision 1: Search is a product feature, not a backend detail</strong></h2><p>The first fork: <strong>how do you find videos?</strong></p><p>You have options:</p><ul><li>YouTube Data API (structured, quota rules, extra auth story)</li><li>Scraping (fast to start, high-maintenance, questionable vibes)</li><li>“Search powered by Google” (good relevance, simple mental model)</li></ul><p>This project uses <strong>Google Programmable Search Engine</strong> (PSE) API because:</p><ul><li><strong>Relevance is the whole game</strong>: for shopping queries, Google’s ranking usually beats anything you can DIY in a weekend.</li><li><strong>Control is “good enough”</strong>: you can restrict results to `youtube.com` and further tweak the engine by user demographics - location, age, language, etc., if you wish to utilize those.</li><li><strong>The UI stays simple</strong>: a text box, results, pick what you want.<br>If you’re new to PSE, start here: G<a href="https://programmablesearchengine.google.com/about/" rel="noreferrer">oogle Programmable Search Engine overview</a>.</li></ul><h3 id="pros"><strong>Pros</strong></h3><p>Great general relevance and “it just works” UX. You can constrain to a site (YouTube) without building search infra.</p><h3 id="cons">Cons</h3><p>Result metadata isn’t as rich as a dedicated YouTube API model. You cannot grab comments, channel details, etc using the search API alone. But hey, that's a problem for another day.</p><h3 id="how-to-pick">How to pick</h3><ul><li>Choose <strong>YouTube Data API</strong> if you need: channels, playlists, publish dates, analytics-ish metadata, or a “video-first” product where precision beats convenience.</li><li>Choose <strong>PSE</strong> if you want: fast iteration, broad query coverage, and you can live with “I guide others to treasures I cannot possess” (I guide others to results I cannot produce/store).</li></ul><h2 id="decision-2-%E2%80%9Ctranscript%E2%80%9D-doesn%E2%80%99t-have-to-mean-%E2%80%9Cverbatim%E2%80%9D">Decision 2: “Transcript” doesn’t have to mean “verbatim”</h2><p>The next fork: <strong>how do you get the content of the video into a model?</strong></p><p>You could chase perfect transcripts. Or you can treat transcripts as a means to an end: <strong>retrieval and citation</strong>.</p><p>YT Shopping AI uses Gemini to generate a <strong>timestamped transcript summary</strong>.</p><p>That’s a deliberate trade:</p><ul><li>The UI needs “good-enough grounding” to answer questions.</li><li>The user needs <strong>source points</strong> (timestamps) more than raw text walls.</li></ul><h3 id="pros-1">Pros</h3><ul><li>It’s fast to read and useful for Q&amp;A.</li><li>Timestamped sections make citations feel real (“go to 08:12”) instead of vibes.</li></ul><h3 id="cons-1">Cons</h3><ul><li>It’s not verbatim; nuance can get compressed.</li><li>Models can hallucinate if the input is weak or the prompt is too confident.</li></ul><h3 id="how-to-pick-1">How to pick</h3><ul><li>If you’re building <strong>compliance</strong> tooling (legal, medical, finance): you want verbatim transcripts and strict provenance.</li><li>If you’re building <strong>consumer decision support</strong>: summaries with timestamps are often the sweet spot.</li></ul><h2 id="decision-3-multi-video-is-the-antidote-to-creator-bias"><strong>Decision 3: Multi-video is the antidote to creator bias</strong></h2><p>Single-video assistants feel smart until you realize they’ve just become a fan account for whoever you watched last.</p><p>We force a more honest workflow:</p><ul><li>search broadly</li><li>pick multiple videos</li><li>answer questions across them</li></ul><p>The structure matters because shopping decisions are inherently comparative:</p><ul><li>“Battery life” isn’t a number; it’s a trade vs weight, thermals, price.</li><li>“Best” changes when you add constraints: budget, region, availability, returns.</li></ul><h3 id="pros-2">Pros</h3><ul><li>You get triangulation: overlap across creators is usually the truth.</li><li>You can surface disagreements explicitly (“Video A says X, Video B says Y”).</li><li>You get over the bias that could have been introduced by sponsored videos.</li></ul><h3 id="cons-2">Cons</h3><ul><li>More compute, more latency.</li><li>More chances for contradiction, which means you need better prompting and UX.</li></ul><h3 id="how-to-pick-2">How to pick</h3><ul><li>If you’re building a “video companion” product (one creator, one long tutorial), single-source is simpler and often better.</li><li>If you’re building “what should I buy?”, multi-source is almost mandatory.</li></ul><h2 id="decision-4-separate-%E2%80%9Ccourse-outline%E2%80%9D-from-%E2%80%9Cqa%E2%80%9D">Decision 4: Separate “course outline” from “Q&amp;A”</h2><p>Don’t ask the model to do everything in one shot.</p><p>The app splits:</p><ul><li>Outline generation: produce a stable structure (“Before you buy”, “Alternatives”, “Reviews”…)</li><li>Q&amp;A: answer targeted questions with sources</li></ul><h3 id="why">Why?</h3><ul><li>Outlines anchor the session. They reduce the “blank page” problem.</li><li>Q&amp;A becomes more surgical. Users ask better questions when they see a map.</li></ul><h3 id="pros-3"><strong>Pros</strong></h3><ul><li>Better UX: users feel guided, not dumped into a chat void.</li><li>Better model behavior: each call has a tight, obvious job.</li></ul><h3 id="cons-3"><strong>Cons</strong></h3><ul><li>More endpoints, more moving pieces.</li><li>You have to reconcile when the outline suggests topics not covered in the selected videos.</li></ul><h2 id="architecture">Architecture</h2><p>Here’s the big picture:</p><figure class="kg-card kg-image-card"><img loading="lazy" src="https://digitalpress.fra1.cdn.digitaloceanspaces.com/4ml9m8u/2025/12/diagram.png" class="kg-image" alt="" loading="lazy" width="1200" height="720"></figure><h3 id="ui-layer">UI layer</h3><p>The UI drives the workflow:</p><ul><li><strong>Search</strong>: query → results</li><li><strong>Select</strong>: choose videos worth trusting</li><li><strong>Extract</strong>: generate transcript-like summaries with timestamps</li><li><strong>Ask</strong>: ask questions across selected videos, get sources back</li><li><strong>Generate</strong>: create a structured “course-like” outline for the shopping topic</li></ul><h3 id="api-routes">API routes</h3><p>The server is intentionally “dumb”: it turns requests into upstream calls.</p><p>Endpoints:</p><ul><li><code>/api/search-videos</code>: calls Google’s Custom Search JSON API endpoint (powered by your PSE CX)</li><li><code>/api/get-transcript</code>: uses Gemini to extract a detailed, timestamped transcript summary from a video URL</li><li><code>/api/ask-videos</code>: uses Gemini to answer questions and returns source timestamps</li><li><code>/api/generate-course</code>: uses Gemini to generate the shopping “course outline” JSON</li></ul><h2 id="a-note-on-keys">A note on keys</h2><p>Keys are stored in the browser and sent per request.
