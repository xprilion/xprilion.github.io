16:T10d3,<p>This post documents the end-to-end process of deploying an autoscaling LLM inference endpoint on Google Kubernetes Engine using Cloud TPU v5e and vLLM. It covers everything from quota discovery to a live, serving deployment – including the parts that didn't go as planned.</p>
<p>This is an abridged version. The full article, all Kubernetes manifests, and helper scripts are available in the <a href="https://github.com/xprilion/gemma3-vllm-tpu-gke-autoscaling" rel="noopener noreferrer">GitHub repository</a>. The detailed writeup is published at <a href="https://xprilion.com/gemma3-vllm-tpu-gke-autoscaling/" rel="noopener noreferrer">xprilion.com/gemma3-vllm-tpu-gke-autoscaling</a>.</p>
<h2>The Goal</h2>
<p>Deploy Gemma 4 on TPU v5e with HPA-based autoscaling driven by vLLM's built-in Prometheus metrics. The autoscaling stack uses Google Cloud Managed Prometheus, the Custom Metrics Stackdriver Adapter, and a HorizontalPodAutoscaler targeting <code>num_requests_waiting</code>.</p>
<h2>What Actually Happened</h2>
<p>TPU provisioning on GCP has three independent gates: regional TPU quota, a global accelerator cap called <code>GPUS_ALL_REGIONS</code>, and physical capacity. I had regional quota (16 chips) but <code>GPUS_ALL_REGIONS</code> was set to 0 – and despite the name, it blocks TPUs too, counted per chip. After testing six zones across three regions and hitting capacity exhaustion everywhere, the quota gate finally revealed itself once capacity opened up.</p>
<p>I requested 16 chips. Google's quota team offered 1. Anything beyond that requires Sales engagement. With a single chip (16GB HBM), I deployed Gemma 3 4B instead of the originally planned Gemma 4 26B.
<img src="https://digitalpress.fra1.cdn.digitaloceanspaces.com/4ml9m8u/2026/04/Screenshot-2026-04-29-at-11.01.55---AM.png" alt="" />GKE Console showing the TPU cluster with both node pools healthy</p>
<h2>Gemma 4 Compatibility</h2>
<p>I tested every Gemma 4 variant available on vLLM's TPU backend (both <code>latest</code> and <code>nightly</code> images). All four attempts failed. The stable image didn't recognize the <code>gemma4</code> architecture. The nightly image got further but hit <code>AssertionError: Expect no shared layers</code> – Gemma 4 uses weight-tied layers that the <code>tpu-inference</code> backend doesn't support yet. This affects all Gemma 4 variants (E2B, E4B, 26B-A4B, 31B).</p>
<h2>The Working Deployment</h2>
<p>The final stack runs Gemma 3 4B on a single TPU v5e chip via <code>vllm/vllm-tpu:nightly</code>, with the full autoscaling infrastructure proven and operational. The HPA reads <code>num_requests_waiting</code> from Managed Prometheus and can scale the deployment from 1 to N replicas, which in turn triggers the GKE node pool autoscaler to provision additional TPU nodes.
<img src="https://digitalpress.fra1.cdn.digitaloceanspaces.com/4ml9m8u/2026/04/Screenshot-2026-04-29-at-11.03.45---AM.png" alt="" />HPA output showing the metric pipeline working: 0/5 targets, 1 replica
The architecture scales to 8 chips and larger models by changing five configuration values: machine type, topology label, TPU count, tensor parallel size, and model name. Everything else - GCS FUSE caching, Workload Identity, PodMonitoring, HPA, load testing scripts – stays identical.</p>
<h2>Conclusion</h2>
<ul>
<li><code>GPUS_ALL_REGIONS</code> is the hidden gate. Check it before anything else.</li>
<li>TPU capacity errors auto-retry; quota errors are permanent. Know which you're hitting.</li>
<li>GKE requires both <code>gke-tpu-accelerator</code> and <code>gke-tpu-topology</code> node selectors. Missing either triggers a Warden rejection.</li>
<li>Build the full stack on minimal hardware first. Prove it works, then scale up.</li>
</ul>
